# ts-git-commit

**何をするか**: 変更をコンベンショナルコミット形式（`type(scope): subject`）でコミットする。
ブランチ保護チェック・変更確認・ユーザー確認の 3 段階ゲートを経て実行する。

**起動フレーズ**: `コミットして` / `変更をコミットして` / `Commit these changes`

---

## ワークフロー

1. **ブランチ確認** — `main`/`master` への直接コミットをブロック。該当する場合は `ts-git-branch` スキルを呼び出してブランチを切り直す。
2. **変更確認** — `git status` / `git diff` で変更内容を把握し、論理的な単位でコミットを分割する計画を立てる。
3. **計画提示** — コミットメッセージと対象ファイルをユーザーに提示して `[yes / edit / cancel]` で確認を取る。
4. **実行** — yes の確認後に `git add` → `git commit` を順番に実行する。

## コミット分割ルール

| 状況 | 判断 |
|---|---|
| 同じ目的の変更（複数ファイル） | 1コミットにまとめる |
| 同じファイルに異なる目的の変更 | コミットを分割する |

## コミットメッセージ形式

```
type(scope): subject
```

- **type**: `feat` / `fix` / `docs` / `style` / `refactor` / `perf` / `test` / `chore`
- **scope**: モジュール名・ディレクトリ名（省略可）
- **subject**: 命令形・英語・30文字以内・ピリオドなし

## 注意事項

- `git push` / `git rebase` / `git merge` は対象外
- `--amend` は使わない（新規コミットを作る）
- pre-commit フックが失敗した場合は手動で修正してから再実行
